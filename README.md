# NDM-test

## Структура проекта

Проект состоит из 3 Nginx, работающих в режиме proxy, и приложения на Python. Приложение слушает HTTP запросы на порту 8080. Приложение в качестве ответа отправляет клиенту содержимое заголовка X-Forwarded-For и ip адресс прокси, от которого приложение непосредственно получило запрос. Прокси и приложение запускаются в отдельных контейнерах. На хост для прокси 1-3 проброшены порты 81-83 соответственно. Приложение на хост не проброшено. Прокси перенаправляют запрос приложению или друг другу, если приложение недоступно напрямую.

Ниже схематично изображена структура проекта

```text

 81                  82                  83
 ============================================
   |                 |                   |
   |    /------------+--------------\    |    
   |   /             |               \   |            
+=====+           +=====+             +=====+
|     |           |     |             |     |
|  1  |-----------|  2  |-------------|  3  |
|     |           |     |             |     |
+=====+           +=====+             +=====+
       \             |               /
        \         +=====+           /
         \        |     |          /
          \-------| app |---------/
                  |     |
                  +=====+

```

## Запуск проекта

```
git clone https://github.com/ast-nt/NDM-test
cd NDM-test
docker compose up -d
```

## Тесты

Запустим команду 
```
curl localhost:81
```

В ответе получим: 
```
root@debian:~# curl localhost:81
192.168.200.1, 192.168.200.101
```
Здесь адрес 192.168.200.1 - адрес клиента (выступает в роли NAT), а 192.168.200.101 - адрес прокси сервера, через который прошел запрос. Последний ip адрес в строке - адрес сервера, от которого приложение непосредственно получило запрос.

Попробуем отправить запрос с поддельным заголовком X-Forwarded-For.
Отправим запрос с заведомо ложным адресом, добавленным клиентом.  
```
curl --header "X-Forwarded-For: 8.8.8.8" localhost:81
``` 
В ответе полуим:  
```
root@debian:~# curl --header "X-Forwarded-For: 8.8.8.8" localhost:81
192.168.200.1, 192.168.200.101
```
Как видим, фальшивый заголовок, высланный клиентом, был проигнорирован.

Аналогичные тесты можно провести для портов 82 и 83, которые прослушиваются прокси 2 и 3 соответственно.

Сэмулируем ситуацию, при которой приложение окажется недоступным для прокси 1 напрямую. Для этого воспользуемся правилами iptables.
Введем команду, которая заблокирует входящие запросы от прокси 1 на приложение.: 
```
docker exec --user root ndm-test-app.local-1 iptables -I INPUT -s 192.168.200.101 -j DROP
```
Теперь еще раз введем команду: 
```
curl --header "X-Forwarded-For: 8.8.8.8" localhost:81
```
В ответе получим:
```
root@debian:~# curl --header "X-Forwarded-For: 8.8.8.8" localhost:81
192.168.200.1, 192.168.200.101, 192.168.200.102
```
В ответе появился еще один ip адрес прокси 2. Прокси 1 послал запрос через него, когда не смог достучаться до приложения напрямую.
